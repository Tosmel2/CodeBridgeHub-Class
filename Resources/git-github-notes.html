What is Git?

Git is a version control system (VCS) — software that tracks changes to files over time. Think of it as an incredibly detailed "undo history" for your entire project, one that lets multiple people work together without overwriting each other's work.

Created by Linus Torvalds in 2005 (the same person who made Linux), Git is now used by virtually every professional software team on the planet.

Why do you need it?

  • Safety net — You can always roll back to any previous state of your project.
  • Collaboration — Multiple developers can work on the same codebase simultaneously.
  • History — You get a permanent log of who changed what, and why.
  • Experimentation — Try risky features in a "branch" without breaking the working version.
  • Industry standard — Nearly every tech job expects you to know Git.

Git vs. GitHub — They are NOT the same thing

ℹ️
Key Distinction

Git is the tool that runs on your computer and tracks changes locally. GitHub is a website that hosts Git repositories online so teams can share and collaborate. Git can exist without GitHub; GitHub is built on top of Git.

GitGitHub
Local software on your machineCloud-based website (github.com)

Installation & Setup

Installing Git

macOS — Git usually comes pre-installed. Check by opening Terminal and typing git --version. If not installed, macOS will prompt you. Or install via Homebrew:

$ brew install git

Windows — Download the installer from git-scm.com. This also installs Git Bash, a terminal emulator.

Linux (Ubuntu/Debian)

$ sudo apt update
$ sudo apt install git

First-time configuration

Before using Git, tell it who you are. This identity gets attached to every commit you make:

# Set your name
$ git config --global user.name "Your Name"

# Set your email (use the same one as your GitHub account)
$ git config --global user.email "you@example.com"

# Set a default branch name (modern standard)
$ git config --global init.defaultBranch main

# Verify your settings
$ git config --list
✅
Tip — Global vs Local config

The --global flag applies settings to all repositories on your machine. Omit it to apply settings to only the current repository.

Core Concepts

Understanding Git's mental model is more important than memorising commands. Here are the fundamental ideas:

Repository (Repo)

A repository is a folder whose contents are tracked by Git. It contains all your project files plus a hidden .git/ folder that stores the entire history. You'll never need to touch .git/ directly — Git manages it.

The Three Areas

Git thinks about your files in three distinct zones:

Zone 1Working Directory
──git add──▶
Zone 2Staging Area
──git commit──▶
Zone 3Repository (History)
  • Working Directory — Where you actually edit files. Changes here are "unsaved" from Git's perspective.
  • Staging Area (Index) — A preparation area. You selectively add changes here before committing. Think of it as assembling items for a photo before taking it.
  • Repository / History — The permanent record. Once committed, changes live here forever (unless explicitly rewritten).

Commit

A commit is a snapshot of your staged changes saved into the repository's history. Each commit has:

  • A unique 40-character identifier called a SHA hash (e.g., a3f8c12...)
  • The author's name and email
  • A timestamp
  • A commit message describing the change
  • A pointer to the previous commit (its "parent")

Branch

A branch is an independent line of development. The default branch is called main (older repos may use master). You can create branches to work on features or fixes in isolation, then merge them back when ready.

HEAD

HEAD is simply a pointer that indicates where you currently are in the repository — usually pointing to the tip (latest commit) of the current branch.

File Status Lifecycle

Untracked ──git add──▶ Staged ──git commit──▶ Committed (Unmodified)
                                                      │
                              Modified ◀──(edit file)─┘
                                  │
                              git add ──▶ Staged ──▶ Committed

Your First Repository

Initialising a new repo

Navigate to your project folder and run:

$ mkdir my-project      # create a folder
$ cd my-project         # enter it
$ git init              # initialise Git here
# → Initialized empty Git repository in .../my-project/.git/

Cloning an existing repo

To download a repository from GitHub onto your machine:

$ git clone https://github.com/username/repo-name.git

# Clone into a custom folder name:
$ git clone https://github.com/username/repo-name.git my-folder

Checking status

Your most-used command. Run it constantly — it tells you exactly what's going on:

$ git status

Output tells you: which branch you're on, which files are modified, staged, or untracked.

The .gitignore file

Create a file named .gitignore at the root of your project to tell Git which files to never track:

# .gitignore
node_modules/       # dependency folders
.env                # secret credentials
*.log               # log files
dist/               # build output
.DS_Store           # macOS system file
✅
Tip

GitHub provides .gitignore templates for every language and framework at github.com/github/gitignore.

Staging & Committing

Staging changes

# Stage a specific file
$ git add index.html

# Stage multiple files
$ git add index.html style.css

# Stage all changes in the current directory
$ git add .

# Stage parts of a file interactively (advanced)
$ git add -p index.html

Making a commit

# Commit with a message (opens default editor if -m is omitted)
$ git commit -m "Add navigation menu to header"

# Stage all tracked files and commit in one step
$ git commit -am "Fix typo in homepage"

Writing good commit messages

A good commit message is concise, written in the imperative mood, and describes what the change does (not how).

❌ Bad✅ Good
stuffAdd user login form
fixed bugFix null pointer error in cart checkout
changesUpdate README with installation steps
wipImplement dark mode toggle (WIP)

Viewing history

# Full log
$ git log

# Compact one-line view
$ git log --oneline

# Visual branch graph
$ git log --oneline --graph --all

# Show changes introduced by each commit
$ git log -p

Viewing differences

# Unstaged changes (working dir vs last commit)
$ git diff

# Staged changes (what will be committed)
$ git diff --staged

Branching

Branches let you diverge from the main line of development and work independently. The golden rule: never commit directly to main on shared projects — always create a branch first.

Branch commands

# List all local branches (* marks current)
$ git branch

# List all branches including remote
$ git branch -a

# Create a new branch
$ git branch feature/login

# Switch to a branch
$ git checkout feature/login

# Create AND switch in one command (modern shorthand)
$ git switch -c feature/signup

# Rename current branch
$ git branch -m new-name

# Delete a branch (after merging)
$ git branch -d feature/login

# Force-delete an unmerged branch
$ git branch -D feature/old

How branching works visually

  main:      A ── B ── C ─────────────── G (merge)
                        \                /
  feature:               D ── E ── F ──

Each letter is a commit. The feature branch diverged at C and is later merged back into main at G.

✅
Naming conventions

Use descriptive, lower-case, hyphenated names. Common patterns: feature/user-auth, fix/login-bug, docs/readme-update, hotfix/payment-crash.

Merging & Conflicts

Merging a branch

To bring changes from another branch into your current branch:

# Switch to the branch you want to merge INTO
$ git checkout main

# Merge the feature branch into main
$ git merge feature/login

If main hasn't moved since the branch was created, Git performs a fast-forward merge (just moves the pointer). Otherwise, Git creates a merge commit.

Rebasing (alternative to merge)

Rebase replays your commits on top of another branch for a cleaner, linear history:

$ git checkout feature/login
$ git rebase main
⚠️
Golden Rule of Rebasing

Never rebase commits that have been pushed to a shared remote repository — it rewrites history and will cause problems for your teammates.

Merge conflicts

A conflict occurs when two branches modify the same line of the same file differently. Git cannot auto-decide which version to keep, so it asks you to resolve it manually.

A conflicted file looks like this:

<<<<<<< HEAD (your current branch)
background-color: blue;
=======
background-color: red;
>>>>>>> feature/redesign (incoming branch)

To resolve:

  1. Open the file and decide which version to keep (or combine them).
  2. Delete all the conflict markers (<<<<<<<, =======, >>>>>>>).
  3. Stage the resolved file: git add style.css
  4. Complete the merge: git commit
# Check which files are conflicted
$ git status

# Abort a merge and start over
$ git merge --abort

GitHub Essentials

Creating an account & your first repo

  1. Sign up at github.com.
  2. Click the + icon → New repository.
  3. Choose a name, set it to Public or Private, optionally add a README.
  4. Copy the HTTPS or SSH URL shown on the repo page.

Key GitHub concepts

TermWhat it means
RepositoryYour project's home on GitHub — code, issues, history
ForkA personal copy of someone else's repository on your account
Pull Request (PR)A request to merge your branch into another branch; used for code review
IssueA task, bug report, or feature request filed on a repo
StarBookmark a repo you like (also a social signal)
WatchGet notifications when things change in a repo
ActionsAutomated workflows (CI/CD) triggered by events like pushes
README.mdThe landing page of a repo, written in Markdown

Pull Requests — the collaboration heartbeat

A PR is how changes get reviewed and merged in a professional workflow:

  1. Create a branch locally and make commits.
  2. Push the branch to GitHub.
  3. Open a Pull Request from that branch into main.
  4. Teammates review, comment, request changes.
  5. Once approved, the PR is merged.

SSH vs HTTPS authentication

GitHub supports two ways to authenticate from the terminal:

  • HTTPS — Easier to set up; uses your GitHub credentials or a Personal Access Token.
  • SSH — More secure and convenient for daily use; requires setting up an SSH key pair once.
# Generate an SSH key (run once)
$ ssh-keygen -t ed25519 -C "you@example.com"

# Copy the public key and paste it into GitHub → Settings → SSH Keys
$ cat ~/.ssh/id_ed25519.pub

# Test the connection
$ ssh -T git@github.com

Remote Repositories

A remote is a version of your repository hosted somewhere else (e.g., GitHub). The default remote is called origin.

Remote commands

# View remotes
$ git remote -v

# Add a remote (after creating a repo on GitHub)
$ git remote add origin https://github.com/you/project.git

# Change a remote URL
$ git remote set-url origin git@github.com:you/project.git

# Remove a remote
$ git remote remove origin

Push — sending commits to GitHub

# Push current branch to origin (first time: set upstream)
$ git push -u origin main

# Subsequent pushes (upstream already set)
$ git push

# Push a feature branch
$ git push origin feature/login

Pull — getting changes from GitHub

# Pull = fetch + merge (most common)
$ git pull

# Pull with rebase instead of merge
$ git pull --rebase

Fetch — checking without merging

# Download remote changes without applying them
$ git fetch origin

# Then inspect what changed
$ git diff main origin/main
ℹ️
Fetch vs Pull

git fetch downloads remote commits but does NOT update your local branch. git pull fetches AND merges in one step. When in doubt, fetch first to inspect, then merge manually.

Typical Workflow

Here's the standard workflow used by most teams — called GitHub Flow:

# 1. Start fresh — pull latest main
$ git checkout main
$ git pull

# 2. Create a feature branch
$ git switch -c feature/add-search

# 3. Make changes, stage, and commit (repeat)
$ ... edit files ...
$ git add .
$ git commit -m "Add search bar to navbar"

# 4. Push branch to GitHub
$ git push -u origin feature/add-search

# 5. Open a Pull Request on GitHub (via browser)
# 6. Get code review → address feedback → approve

# 7. Merge the PR on GitHub (or via CLI)
$ git checkout main
$ git merge feature/add-search

# 8. Delete the feature branch
$ git branch -d feature/add-search
$ git push origin --delete feature/add-search

Undoing Mistakes

One of Git's superpowers. Here's how to undo different kinds of mistakes:

Unstage a file (before committing)

$ git restore --staged index.html

Discard changes in working directory

# ⚠️ Irreversible — discards unsaved changes to a file
$ git restore index.html

# Discard ALL uncommitted changes
$ git restore .

Amend the last commit

# Fix the commit message (before pushing)
$ git commit --amend -m "Corrected commit message"

# Add a forgotten file to the last commit
$ git add forgotten-file.js
$ git commit --amend --no-edit

Revert a commit (safe — creates new commit)

# Creates a new commit that undoes the specified commit
# Safe to use even on pushed commits
$ git revert a3f8c12

Reset (powerful — rewrites history)

# Soft reset: undo commits, keep changes staged
$ git reset --soft HEAD~1

# Mixed reset (default): undo commits, keep files in working dir
$ git reset HEAD~1

# Hard reset: undo commits AND discard all changes ⚠️
$ git reset --hard HEAD~1
⚠️
Never force-push to shared branches

git reset --hard followed by git push --force on a shared branch will overwrite teammates' work. Only use force-push on branches only you are working on.

Stashing — temporary storage

# Save uncommitted changes to a stash (cleans working dir)
$ git stash

# View stashes
$ git stash list

# Re-apply most recent stash
$ git stash pop

# Apply a specific stash
$ git stash apply stash@{2}

# Drop a stash
$ git stash drop stash@{0}
